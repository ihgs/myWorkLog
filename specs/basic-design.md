# 基本設計書 — Work Log Management (mylog)

## 1. 概要

仕事の作業量（工数）を管理するデスクトップアプリケーション。
作業区分ごとの予定工数に対して日々の実績工数を記録し、ダッシュボードで予定と実績を可視化する。

- **対象プラットフォーム**: macOS / Windows
- **データ永続化**: SQLite（ローカルファイル）
- **工数・時間の単位**: すべて「時間(h)」で統一する。基準線デフォルト `8` は「1日8時間」を意味する。

---

## 2. EARS記法について

本書の要求は **EARS（Easy Approach to Requirements Syntax）** で記述する。各要求は次のいずれかのパターンに従う。

| パターン | 構文 | 用途 |
|----------|------|------|
| ユビキタス（常時） | 本システムは、〜しなければならない | 常に成立する要求 |
| イベント駆動 | 〜したとき、本システムは、〜しなければならない | 特定操作・事象への応答 |
| 状態駆動 | 〜の間、本システムは、〜しなければならない | ある状態の間だけ成立 |
| オプション | 〜の場合、本システムは、〜しなければならない | 任意機能・条件付き |
| 望ましくない動作 | もし〜ならば、本システムは、〜しなければならない | 異常・例外への対応 |

要求IDは `R-<機能>-<連番>` 形式で付与する。

### 3.1 アーキテクチャ要求

- **R-ARCH-1**（ユビキタス）: 本システムは、すべてのデータベースアクセスをRust側に集約しなければならない。
- **R-ARCH-2**（ユビキタス）: 本システムは、フロントエンドからのデータ取得・更新をTauriコマンド（`invoke`）経由で行わなければならない。
- **R-ARCH-3**（ユビキタス）: 本システムは、SQL文をフロントエンドに記述してはならない。
- **R-ARCH-4**（ユビキタス）: 本システムは、工数・時間の集計ロジックをRust側に実装しなければならない。

## 4. データモデル

### 4.1 ER概要

```
WorkCategory (作業区分) 1 ── * MonthlyPlan (月予定)
WorkCategory (作業区分) 1 ── * ActualWork (実績工数)
Setting (設定) … 単一レコード（基準線など）
```

### 4.2 エンティティ定義

テーブル・カラムの詳細定義（型・制約・説明・リレーション）は、別ファイル **[`data-model.dbml`](./data-model.dbml)** に DBML 形式で記載する。各テーブル・カラムの説明は DBML 内の `Note` として埋め込んでいる。

対象テーブル:

| テーブル | 役割 |
|----------|------|
| `work_category` | 作業区分。月予定・実績工数の親エンティティ |
| `monthly_plan` | 月予定。作業区分ごと・対象月ごとの予定工数 |
| `actual_work` | 実績工数。日々の作業実績 |
| `setting` | アプリ設定（基準線など）。単一レコード |

### 4.3 データ永続化の要求

- **R-DATA-1**（ユビキタス）: 本システムは、すべての業務データをSQLiteに永続化しなければならない。
- **R-DATA-2**（イベント駆動）: 初回起動したとき、本システムは、テーブルを作成し `setting` の初期レコード（`baseline_hours = 8`）を投入しなければならない。
- **R-DATA-3**（ユビキタス）: 本システムは、SQLiteのDBファイルをアプリデータディレクトリ（`app_data_dir`）配下に配置しなければならない。
- **R-DATA-4**（イベント駆動）: 作業区分が削除されたとき、本システムは、当該作業区分に紐づく月予定および実績工数をすべて削除しなければならない（カスケード削除）。

## 5. 機能要求（EARS）

### 5.1 作業区分管理（F1）

- **R-CAT-1**（イベント駆動）: ユーザーが作業区分の登録を確定したとき、本システムは、コード・名前・予定工数・月予定（複数可）を保存しなければならない。
- **R-CAT-2**（ユビキタス）: 本システムは、コードの重複登録を許容しなければならない。
- **R-CAT-3**（イベント駆動）: ユーザーが作業区分一覧を開いたとき、本システムは、登録済みの全作業区分を一覧表示しなければならない。
- **R-CAT-4**（イベント駆動）: ユーザーが作業区分の編集を確定したとき、本システムは、対象の作業区分の内容（月予定含む）を更新しなければならない。
- **R-CAT-5**（イベント駆動）: ユーザーが作業区分の削除を実行したとき、本システムは、対象の作業区分を削除しなければならない。
- **R-CAT-6**（イベント駆動）: ユーザーが作業区分の削除を要求したとき、本システムは、関連する実績工数も削除される旨を確認ダイアログで提示しなければならない。
- **R-CAT-7**（望ましくない動作）: もし必須項目（コード・名前）が未入力ならば、本システムは、登録・編集を拒否しエラーを通知しなければならない。
- **R-CAT-8**（望ましくない動作）: もし予定工数が負数ならば、本システムは、登録・編集を拒否しエラーを通知しなければならない。
- **R-CAT-9**（オプション）: 月予定を入力する場合、本システムは、対象月を `yyyy/mm` 形式で受け付けなければならない。

### 5.2 実績工数入力（F2）

- **R-ACT-1**（イベント駆動）: ユーザーが実績工数の登録を確定したとき、本システムは、作業区分・実績時間・作業日・メモを保存しなければならない。
- **R-ACT-2**（ユビキタス）: 本システムは、実績工数の入力時に既存の作業区分から選択させなければならない。
- **R-ACT-3**（イベント駆動）: ユーザーが実績一覧を開いたとき、本システムは、作業日・作業区分・実績時間・メモを一覧表示しなければならない。
- **R-ACT-4**（オプション）: 期間または作業区分による絞り込みが指定された場合、本システムは、条件に一致する実績工数のみを表示しなければならない。
- **R-ACT-5**（イベント駆動）: ユーザーが実績工数の編集を確定したとき、本システムは、対象の実績工数を更新しなければならない。
- **R-ACT-6**（イベント駆動）: ユーザーが実績工数の削除を実行したとき、本システムは、対象の実績工数を削除しなければならない。
- **R-ACT-7**（望ましくない動作）: もし実績時間が未入力または負数ならば、本システムは、登録・編集を拒否しエラーを通知しなければならない。
- **R-ACT-8**（ユビキタス）: 本システムは、作業日を `yyyy/mm/dd` 形式で保持しなければならない。
- **R-ACT-9**（ユビキタス）: 本システムは、メモを任意入力として扱わなければならない。

### 5.3 ダッシュボード（F3）

#### 共通

- **R-DASH-1**（イベント駆動）: ダッシュボードを開いたとき、本システムは、当月を初期表示の対象月としなければならない。
- **R-DASH-2**（イベント駆動）: ユーザーが対象月を変更したとき、本システムは、選択された月のデータでグラフを再描画しなければならない。

#### グラフ1: 予定工数 vs 実績工数（棒グラフ）

- **R-DASH-3**（ユビキタス）: 本システムは、対象月の予定工数と実績工数を棒グラフで表示しなければならない。
- **R-DASH-4**（ユビキタス）: 本システムは、予定工数と実績工数を「全体」および「各作業区分ごと」の両方で表示しなければならない。
- **R-DASH-5**（ユビキタス）: 本システムは、対象月の予定工数として `MonthlyPlan`（月予定）のみを使用しなければならない。
- **R-DASH-6**（オプション）: 対象月の月予定が存在しない作業区分の場合、本システムは、その月の予定工数を `0` として扱わなければならない。

#### グラフ2: 日別 実績時間 積み上げ棒グラフ

- **R-DASH-7**（ユビキタス）: 本システムは、対象月の1日ごとの実績時間を積み上げ棒グラフで表示しなければならない。
- **R-DASH-8**（ユビキタス）: 本システムは、積み上げ棒の各セグメントを作業区分ごとに色分けして表示しなければならない。
- **R-DASH-9**（ユビキタス）: 本システムは、基準線を横線でグラフ上に表示しなければならない。
- **R-DASH-10**（ユビキタス）: 本システムは、各日の棒の上にその日の合計時間を表示しなければならない。
- **R-DASH-11**（オプション）: 実績データが存在しない日の場合、本システムは、当該日を1日から月末までの軸上に表示しなければならない。

### 5.4 設定（F4）

- **R-SET-1**（イベント駆動）: 設定画面を開いたとき、本システムは、現在の基準線（`baseline_hours`）を表示しなければならない。
- **R-SET-2**（イベント駆動）: ユーザーが基準線を更新したとき、本システムは、新しい基準線を保存しなければならない。
- **R-SET-3**（ユビキタス）: 本システムは、基準線の初期値を `8` としなければならない。
- **R-SET-4**（イベント駆動）: 基準線が更新されたとき、本システムは、日別積み上げグラフの基準線に新しい値を反映しなければならない。

## 7. Tauriコマンド（API）設計

すべて非同期、戻り値は `Result<T, String>`（エラーは文字列メッセージ）。

### 7.1 作業区分

| コマンド | 引数 | 戻り値 | 説明 |
|----------|------|--------|------|
| `list_work_categories` | — | `WorkCategory[]`（月予定含む） | 一覧取得 |
| `create_work_category` | `code, name, planned_hours, monthly_plans[]` | `WorkCategory` | 登録 |
| `update_work_category` | `id, code, name, planned_hours, monthly_plans[]` | `WorkCategory` | 編集 |
| `delete_work_category` | `id` | `()` | 削除（関連カスケード削除） |

### 7.2 実績工数

| コマンド | 引数 | 戻り値 | 説明 |
|----------|------|--------|------|
| `list_actual_works` | `from_date?, to_date?, work_category_id?` | `ActualWork[]` | 一覧取得（絞り込み可） |
| `create_actual_work` | `work_category_id, actual_hours, work_date, memo` | `ActualWork` | 登録 |
| `update_actual_work` | `id, work_category_id, actual_hours, work_date, memo` | `ActualWork` | 編集 |
| `delete_actual_work` | `id` | `()` | 削除 |

### 7.3 ダッシュボード集計

| コマンド | 引数 | 戻り値 | 説明 |
|----------|------|--------|------|
| `get_dashboard_summary` | `year_month`（`yyyy/mm`） | `DashboardSummary` | 月単位の予定/実績集計（区分別・全体） |
| `get_daily_stacked` | `year_month`（`yyyy/mm`） | `DailyStacked` | 日別×区分別の実績積み上げデータ |

`DashboardSummary`（イメージ）:
```ts
{
  yearMonth: string;
  total: { plannedHours: number; actualHours: number };
  categories: Array<{
    workCategoryId: number;
    code: string;
    name: string;
    plannedHours: number;   // 当月のmonthly_plan（無ければ0）
    actualHours: number;    // 当月の実績合計
  }>;
}
```

`DailyStacked`（イメージ）:
```ts
{
  yearMonth: string;
  baselineHours: number;          // setting.baseline_hours
  days: Array<{
    date: string;                 // 'yyyy/mm/dd'（1日〜月末まで全て含む）
    totalHours: number;           // その日の合計
    byCategory: Array<{ workCategoryId: number; name: string; hours: number }>;
  }>;
}
```

### 7.4 設定

| コマンド | 引数 | 戻り値 | 説明 |
|----------|------|--------|------|
| `get_setting` | — | `Setting` | 設定取得 |
| `update_setting` | `baseline_hours` | `Setting` | 基準線更新 |

## 8. 画面構成

| 画面 | パス | 概要 |
|------|------|------|
| ダッシュボード | `/` | 予定/実績グラフ、日別積み上げグラフ |
| 作業区分一覧 | `/categories` | 作業区分の一覧・登録・編集・削除 |
| 実績入力 | `/actuals` | 実績工数の一覧・登録・編集・削除 |
| 設定 | `/settings` | 基準線の設定 |

- **R-UI-1**（ユビキタス）: 本システムは、ダッシュボード・作業区分・実績入力・設定の各画面を切り替えるナビゲーションを提供しなければならない。
- **R-UI-2**（ユビキタス）: 本システムは、日付・月の表示を `yyyy/mm` および `yyyy/mm/dd` 形式で行わなければならない。

## 9. 非機能・横断要求

- **R-NF-1**（望ましくない動作）: もしTauriコマンドがエラーを返したならば、本システムは、ユーザーにエラーメッセージを通知しなければならない。
- **R-NF-2**（ユビキタス）: 本システムは、入力値のバリデーション（必須・数値0以上）をフロントエンドとRust側の双方で行わなければならない。
- **R-NF-3**（ユビキタス）: 本システムは、macOSおよびWindowsで動作しなければならない。
- **R-NF-4**（ユビキタス）: 本システムは、集計ロジック（Rust側）に対する単体テストを備えなければならない。

- データのエクスポート/インポート（CSV等）。
- 複数月またぎ・年間ビューなどの拡張ダッシュボード。
