## 目的

EARS（Easy Approach to Requirements Syntax）で書かれた設計書を読み取り、Gherkin記法の振る舞い定義ファイル（`.feature`）を生成する。EARS要求とGherkinシナリオの**双方向トレーサビリティ**を担保する。

## 入力

- 設計書: `$1`（未指定なら `specs/basic-design.md`）
- 出力フォルダ: `$2`（未指定なら `specs/features`）

## 手順

1. 設計書 `$1` を読み、全EARS要求を抽出する。
   - 要求IDは `R-<機能>-<連番>` 形式。各要求のEARSパターン（ユビキタス / イベント駆動 / 状態駆動 / オプション / 望ましくない動作）を把握する。
   - 関連する補助資料（ER図・DBML・コマンド定義・画面構成）も参照し、シナリオの具体値の根拠にする。
2. 出力フォルダ `$2` を作成する（存在すれば再利用）。
3. 機能領域（F1, F2, … や ARCH/DATA/UI/NF などの横断要求）ごとに `.feature` ファイルを分割生成する。
4. 生成した `.feature` とは別に、`$2/traceability.md`（トレーサビリティマトリクス）を生成する。

## Gherkin記述ルール

- 先頭行に `# language: ja` を付け、日本語キーワード（機能 / 背景 / シナリオ / シナリオアウトライン / 前提 / もし / ならば / かつ / 例）で記述する。
- ファイル冒頭コメントに「機能領域」「対応EARS（設計書の節）」「トレーサビリティ方針」を記載する。
- `ならば` 節はEARS要求文（「本システムは〜しなければならない」）に語彙を寄せ、対応関係を読み取れるようにする。
- パターン別の書き分け:
  - **望ましくない動作 / オプション** で複数の入力条件があるものは `シナリオアウトライン` + `例` で網羅する。
  - **ユビキタス**（常時成立）も観測可能なら具体シナリオに落とす。

## トレーサビリティ担保（必須）

1. **タグ方式**: 各シナリオに、対応するEARS要求IDと**完全一致**するタグ `@R-<機能>-<連番>` を付与する。
   - 1シナリオが複数要求を満たす場合は複数タグを付ける。
   - 補助タグ（`@option` / `@異常系` / `@ui` / `@validation` など）も併用可。
2. **マトリクス方式**: `$2/traceability.md` に、**設計書の全要求IDを漏れなく**列挙した対応表を作る。
   - 列: `要求ID | EARSパターン | featureファイル | シナリオ（タグ） | 検証方法`
   - 振る舞いで観測困難な要求（アーキテクチャ制約・対応OS・単体テスト整備など）はGherkin化せず、「検証方法」列に検証手段（コードレビュー / 構成 / CI / 単体テスト等）を明記し `—` でfeature欄を埋める。
   - 末尾にカバレッジ要約（総要求数 / Gherkin検証数 / 非振る舞い検証数）を記載する。

## 完了時の自己検証

- `grep -rho '@R-[A-Z]*-[0-9]*' $2/*.feature | sort -u` でタグ付与済みの要求IDを数え、設計書の要求数・traceability.mdの記載と突き合わせ、過不足を報告する（コメント中のプレースホルダ `@R-XXX-` は除外）。
- 取りこぼした要求IDがあれば追記し、合計数の整合を取る。

## 出力報告

- 生成ファイル一覧、要求カバレッジ（総数 / Gherkin / 非振る舞い）、トレーサビリティの担保方法を簡潔に報告する。
- Gherkinランナー（cucumber/behave等）未導入の場合、構文の実行検証は未実施である旨を明示する。
